# Vibe Coding Golden Rules (Expo/Windows Edition)

## 第1条：【絶対遵守】バージョンの決定権は「AI」ではなく「CLI」にある
AIは2023年や2024年の学習データに基づいて「推測」でバージョンを提案しますが、Expoの最新環境（SDK 54など）とは必ずズレます。

🚫 **禁止**: `package.json` を手動で書き換えること。AIに「バージョンを指定して」と頼むこと。

✅ **正解**: ライブラリ追加は必ず `npx expo install パッケージ名` を使うこと。

**理由**: このコマンドだけが、現在のSDKバージョンに適合する「唯一の正解」をサーバーから取得できます。

## 第2条：環境構築は「減点方式」ではなく「加点方式」で進める
最初から全部盛りのコードをコピペしてエラーが出ると、原因が特定できません。

**ルール**:
1. まず `npx create-expo-app` で「Hello World」が表示されることを確認する。
2. 次に `NativeWind` だけを入れて、色がつくか確認する。
3. 次に `Reanimated` だけを入れて、動くか確認する環境が整ってから初めて、ビジネスロジック（機能）のコードを書き始める。

## 第3条：Windowsユーザーの「清掃義務」
WindowsはmacOSやLinuxに比べて、ファイルロック（使用中のファイルを消せない仕様）やキャッシュの残留が激しいです。

**トラブル時の鉄則**: エラーが出たら「コードを直す」前に、以下の「Windows除霊コマンド」を走らせる。

```powershell
# プロセスを殺し、ゴミを消し、キャッシュを無視して起動
taskkill /IM node.exe /F
Remove-Item -Recurse -Force node_modules
npm cache clean --force
npm install
npx expo start --clear
```

## 第4条：「30分ルール」（損切りラインの設定）
環境エラー（赤画面やビルドエラー）の解決に30分以上かかったら、そのプロジェクトフォルダは「汚染された」とみなして捨てること。

**アクション**: 悩む時間を捨てて、即座に `npx create-expo-app` で新品のプロジェクトを作り、`app` フォルダ（ソースコード）だけを移植する。これが最短の解決策です。

## 第5条：AIへの「プロンプト」の規格化
AIエージェント（CursorやWindsurfなど）に指示を出す際、最初に以下のコンテキストを渡して「釘を刺す」こと。

### 📋 AIエージェント用 プロンプトテンプレート（コピペ用）
新規プロジェクトを開始する際、最初にこのプロンプトをAIに入力してください。

```markdown
# Role
あなたはReact Native (Expo) のエキスパートです。
現在の環境は【Windows 11 / Expo SDK 54 / PowerShell】です。

# Constraints (絶対厳守)
1. ライブラリのインストール指示は、必ず `npm install` ではなく **`npx expo install`** を使用してください。バージョン番号を直接指定 (`@0.76.6`など) することは禁止します。
2. エラー解決の際、`package.json` のバージョン番号を手動で書き換える提案はしないでください。必ずCLIコマンドによる自動解決を優先してください。
3. スタイリングには `NativeWind v2` を使用します。v4の設定を混同しないでください。
4. 私の環境はWindowsです。キャッシュ削除の指示を出す際は、`rm -rf` ではなく PowerShell用のコマンド (`Remove-Item`) を提示してください。

# Philosophy
「推測」でバージョンを提案せず、常に「Expo CLI」にバージョン解決を委ねる安全な手順を提示してください。
```

---

## 今後のフローのイメージ
1. **Project Start**: `npx create-expo-app my-app`
2. **Config**: 上記プロンプトをAIに投げる。
3. **Install**: `npx expo install nativewind` など、1つずつ入れて起動確認。
4. **Coding**: ここから初めて「バイブコーディング（爆速開発）」スタート。
5. **Error**: もし環境エラーが出たら、戦わずに「新品へ移植」。

このルールさえ守れば、今回のような「バージョン地獄」は完全に防げます。
**環境構築は「臆病」に、コーディングは「大胆」に。** これが成功の秘訣です。
